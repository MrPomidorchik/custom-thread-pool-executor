# Custom Thread Pool Executor

Проект представляет собой реализацию собственного пула потоков на Java без использования стандартного `ThreadPoolExecutor`.

Пул поддерживает настройку количества потоков, ограниченные очереди задач, время простоя потоков, минимальное количество резервных потоков, политику отказа и подробное логирование работы.

## Быстрый старт

1. Настройте параметры пула потоков в файлах:

```text
src/main/resources/pool.env
src/main/resources/overload-pool.env

# Доступные политики отказа:
    ABORT
    CALLER_RUNS
```
- файл `pool.env` используется для обычной демонстрации работы пула.

- файл `overload-pool.env` используется для демонстрации перегрузки и политики отказа.

Запустите проект любым удобным способом (через среду разработки или через терминал).

## Основные возможности

- настройка `corePoolSize`;
- настройка `maxPoolSize`;
- настройка `keepAliveTime`;
- настройка размера очереди `queueSize`;
- поддержка `minSpareThreads`;
- выполнение задач через `execute(Runnable)`;
- выполнение задач через `submit(Callable<T>)`;
- получение результата через `Future<T>`;
- мягкое завершение через `shutdown()`;
- принудительное завершение через `shutdownNow()`;
- обработка отказов при перегрузке;
- распределение задач по алгоритму Round Robin;
- логирование создания потоков, выполнения задач, отказов и завершения потоков.

## Анализ производительности

Для сравнения можно рассматривать стандартный `ThreadPoolExecutor` из Java. В стандартной библиотеке пул потоков реализован более оптимизированно: он лучше протестирован, имеет более гибкие механизмы управления очередями, потоками, состояниями завершения и отказами. Поэтому в реальных промышленных системах предпочтительнее использовать именно стандартные реализации.

Собственный пул потоков уступает `ThreadPoolExecutor` по уровню оптимизации и надёжности, но позволяет явно показать внутреннюю механику работы: создание потоков, распределение задач по очередям, завершение потоков по таймауту простоя и обработку перегрузки. Главное преимущество разработанного пула — прозрачность работы. По логам можно отследить, в какую очередь попала задача, какой поток её выполнил, когда был создан новый поток, когда поток завершился по idle timeout и когда задача была отклонена из-за перегрузки.

## Мини-исследование параметров пула

Производительность пула зависит от нескольких параметров:

- `corePoolSize`;
- `maxPoolSize`;
- `queueSize`;
- `keepAliveTime`;
- `minSpareThreads`;
- выбранной политики отказа.

В демонстрационной программе используются два набора параметров.

### Обычный режим работы

Файл:

```text
src/main/resources/pool.env
```

Параметры:

```env
POOL_NAME=MyPool
CORE_POOL_SIZE=2
MAX_POOL_SIZE=4
KEEP_ALIVE_TIME=5
TIME_UNIT=SECONDS
QUEUE_SIZE=5
MIN_SPARE_THREADS=1
REJECTION_POLICY=ABORT
```

Этот набор параметров показывает нормальную работу пула. Базово создаются 2 потока, но при росте нагрузки пул может расшириться до 4 потоков. Значение `MIN_SPARE_THREADS=1` позволяет заранее создавать дополнительный поток, если свободных потоков становится слишком мало.

Такая конфигурация хорошо подходит для демонстрации:

- обработки нескольких задач параллельно;
- создания дополнительных потоков;
- распределения задач по очередям;
- завершения лишних потоков после простоя.

### Режим перегрузки

Файл:

```text
src/main/resources/overload-pool.env
```

Параметры:

```env
POOL_NAME=OverloadPool
CORE_POOL_SIZE=1
MAX_POOL_SIZE=2
KEEP_ALIVE_TIME=3
TIME_UNIT=SECONDS
QUEUE_SIZE=1
MIN_SPARE_THREADS=0
REJECTION_POLICY=ABORT
```

Этот набор параметров специально сделан ограниченным. У пула всего 1 базовый поток, максимум 2 потока и очередь размером 1 задача на каждого Worker. Благодаря этому можно быстро создать ситуацию перегрузки и проверить работу политики отказа `AbortPolicy`.

При такой конфигурации часть задач принимается в работу, а остальные отклоняются с сообщением:

```text
[Rejected] Task ... was rejected due to overload!
```

### Выводы по подбору параметров

Оптимальные параметры зависят от характера задач.

Для коротких CPU-bound задач, которые активно используют процессор, невыгодно создавать слишком много потоков. Если потоков значительно больше, чем ядер процессора, появляются лишние переключения контекста, и производительность может снизиться.

Для I/O-bound задач или задач с ожиданием, например сетевых запросов или операций со `sleep`, количество потоков можно делать больше, так как часть потоков часто находится в ожидании.


## Механизм распределения задач

В проекте реализовано распределение задач по алгоритму Round Robin.

Каждый `Worker` имеет собственную очередь задач. При поступлении новой задачи пул выбирает следующую очередь по кругу:

```text
task-1 -> queue-1
task-2 -> queue-2
task-3 -> queue-3
task-4 -> queue-4
task-5 -> queue-1
```

Если выбранная очередь заполнена, пул пытается добавить задачу в другую доступную очередь. Если все очереди заполнены, но количество потоков ещё меньше `maxPoolSize`, создаётся новый `Worker` со своей очередью. Если создать новый поток уже нельзя, вызывается политика отказа.
